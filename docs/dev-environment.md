# Development environment & secrets

Build sessions run on a VT VPS, accessed remotely through **Orca**. Orca creates a **fresh git worktree per session**, so local env state never survives between sessions — secrets live in **Doppler** and every worktree pulls them on creation.

- **Doppler project:** `agent-deployment-template` (workplace `vt`), configs `dev` / `stg` / `prd`. All credentials for the stack go here via the Doppler CLI (`doppler secrets set NAME --project agent-deployment-template --config dev`); nothing secret is ever committed. Per-deployment clones get their own Doppler project (the runbook's sed step points `worktree-setup.sh` at it).
- **Worktree setup:** `scripts/worktree-setup.sh` — Orca runs it in each new worktree (safe to re-run manually). It binds the directory to the Doppler project and writes the current secrets to a local `.env` (gitignored, mode 600). Override the config with `DOPPLER_CONFIG=stg ./scripts/worktree-setup.sh`.
- **Adding a credential:** set it in Doppler, then re-run the setup script in any worktree that needs it.
- **All dev auth must be headless.** This environment has no browser, and interactive login flows that use a localhost callback cannot complete over SSH. Every vendor CLI/SDK on this box authenticates via a token/API key from Doppler. When a vendor only mints tokens from an interactively-authenticated CLI (Mastra does: `mastra auth login` opens a localhost-callback browser flow, and tokens come from `mastra auth tokens create <name>`), run that login once on a local machine with a browser, mint a long-lived token there, and store it in Doppler (`MASTRA_API_TOKEN`). The VPS then authenticates purely from the env var. (Fallback if installing a CLI locally is undesirable: SSH port-forward the callback port shown in the login URL, e.g. `ssh -L <port>:localhost:<port>`.)
